3. Layer 1 — What triggers a workflow run

The on: block in the workflow YAML defines four triggers:

on:
  push:
    branches: [main]
    paths:
      - 'Project_04_Test_Plan_Creator_Agent_V2/**'
      - '.github/workflows/e2e-tests.yml'
  pull_request:
    branches: [main]
    paths: [same as above]
  schedule:
    - cron: '0 2 * * *'         # 02:00 UTC nightly
  workflow_dispatch:             # manual button
    inputs:
      reason:
        description: 'Why are you running this manually?'
TriggerWhen it fires
pushYou push a commit to main AND at least one changed file is under Project_04_Test_Plan_Creator_Agent_V2/
pull_requestA PR against main with the same path filter
schedule02:00 UTC every day (≈ 07:30 IST). Catches dependency drift
workflow_dispatchYou click "Run workflow" in GitHub Actions UI OR "Trigger Run" in the in-app dashboard

A "concurrency" block then cancels any older run on the same branch when a new push arrives:

concurrency:
  group: e2e-${{ github.ref }}
  cancel-in-progress: true

4. Layer 1 — The three jobs

typecheck ← runs first (Node 20, deps cached) │ ▼ ┌───┴───┐ │ │ build e2e ← run in parallel after typecheck succeeds

Job 1: typecheck

Fast TypeScript validation. If your code doesn't compile, fail here before spending compute on browser tests.

steps:
  - uses: actions/checkout@v4          # clone repo onto runner
  - uses: actions/setup-node@v4        # install Node 20 + npm cache
  - run: npm ci                        # frontend deps
  - run: npx tsc --noEmit              # frontend typecheck
  - run: npm ci                        # backend deps (working-directory: .../backend)
  - run: npx tsc --noEmit              # backend typecheck

cache: 'npm' hashes package-lock.json and reuses node_modules across runs. First run is slow; subsequent runs save ~30–60 seconds.

Job 2: build

Compiles the frontend into dist/ and uploads it as a downloadable artifact (kept 7 days).

needs: typecheck
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
  - run: npm ci
  - run: npm run build                 # tsc -b && vite build
  - uses: actions/upload-artifact@v4
    with:
      name: frontend-dist
      path: Project_04_Test_Plan_Creator_Agent_V2/dist
      retention-days: 7

Job 3: e2e (the interesting one)

Hard 30-minute cap. Sets CI=true so Playwright runs headless.

needs: typecheck
timeout-minutes: 30
env: { CI: 'true' }
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
  - run: npm ci                        # frontend deps
  - run: npm ci                        # backend deps
  - run: npx playwright install --with-deps chromium

  - name: Start backend
    run: |
      (cd backend && npx tsx server.ts) &
      echo "$!" > backend.pid
      for i in $(seq 1 30); do
        if curl -sf http://localhost:3001/api/health > /dev/null; then
          echo "Backend ready after $i attempts"
          exit 0
        fi
        sleep 1
      done
      echo "Backend failed to start"; exit 1

  - name: Run Playwright tests
    run: |
      if find tests -name "*.spec.ts" 2>/dev/null | grep -q .; then
        npx playwright test
      else
        echo "::warning::No Playwright test files found in tests/ — skipping run"
      fi

  - name: Stop backend
    if: always()
    run: |
      if [ -f backend.pid ]; then
        kill "$(cat backend.pid)" 2>/dev/null || true
      fi

  - name: Upload Playwright HTML report
    if: always()                       # upload even on failure
    uses: actions/upload-artifact@v4
    with:
      name: playwright-report
      path: Project_04_Test_Plan_Creator_Agent_V2/playwright-report
      retention-days: 14

  - name: Upload test results (screenshots, videos, traces)
    if: always()
    uses: actions/upload-artifact@v4
    with:
      name: test-results
      path: Project_04_Test_Plan_Creator_Agent_V2/test-results
      retention-days: 14
Why your runs currently print a yellow warning: the find tests -name "*.spec.ts" guard fails because no spec files exist at the project's tests/ root yet. To actually run Playwright tests in CI, commit the generated specs (e.g. from tests/generated/Demo/) into a tracked location and adjust the find path if needed.

5. Layer 2 — The in-app CI/CD dashboard

The app embeds a dashboard so you can see runs and trigger them without leaving the editor.

Frontend (CICDDashboard.tsx) │ │ fetch('http://localhost:3001/api/cicd/...') ▼ Backend (cicd.ts + server.ts endpoints) │ │ axios.get/post('https://api.github.com/...', │ headers: { Authorization: 'Bearer <PAT>' }) ▼ GitHub Actions REST API
Security note: The Personal Access Token never reaches the browser. The frontend hits localhost endpoints; the backend holds the token in backend/cicd-config.json and adds the Authorization header on every GitHub call. That's the whole reason cicd.ts exists as a proxy.

The 7 backend endpoints

Endpoint What it does GitHub API call
GET /api/cicd/configReturn current settings; token is masked(none)
POST /api/cicd/configSave new settings(none)
POST /api/cicd/testVerify token worksGET /user
GET /api/cicd/workflowGet configured workflow's metadataGET /repos/.../actions/workflows/{file}
GET /api/cicd/workflowsList ALL workflows in the repoGET /repos/.../actions/workflows
GET /api/cicd/runs?limit=20Last N runs of the configured workflowGET /repos/.../actions/workflows/{file}/runs
POST /api/cicd/triggerStart a manual runPOST /repos/.../actions/workflows/{file}/dispatches

The "Trigger Run" sequence

  1. Frontend: POST /api/cicd/trigger { ref: "main", reason: "smoke test" }
  2. Backend's triggerWorkflow() reads config, asserts owner/repo/token are set, calls:
    POST https://api.github.com/repos/BalajiS108/AI-Testing-All-About/actions/workflows/e2e-tests.yml/dispatches
    body { ref: "main", inputs: { reason: "smoke test" } }
  3. GitHub responds 204 No Content immediately (accepts the request)
  4. GitHub Actions servers asynchronously match the dispatch against the YAML's on: workflow_dispatch block, allocate a runner, begin execution
  5. Backend returns to frontend: { triggeredAt, branch }
  6. Dashboard polls /api/cicd/runs; the new run appears with status queued → in_progress → completed

Configuration file

// backend/cicd-config.json
{
  "owner": "BalajiS108",
  "repo": "AI-Testing-All-About",
  "token": "ghp_xxxxxxxxxxxxx",
  "workflowFile": "e2e-tests.yml",
  "defaultBranch": "main"
}

When the frontend asks for the config, the backend returns everything except the token — replaced with tokenSet: boolean. The token can never be read back out; only overwritten with a new value.

6. Layer 3 — The local CI mirror

run-ci-locally.sh replays the workflow stages on your machine without a GitHub round-trip. Useful as a pre-flight check before pushing.

npm run ci:local
# or directly:
bash run-ci-locally.sh --skip-install   # faster on re-runs
bash run-ci-locally.sh --skip-e2e       # typecheck + build only

Stages (mirror the YAML)

  1. npm ci (frontend)
  2. cd backend && npm ci
  3. npx tsc --noEmit (frontend)
  4. cd backend && npx tsc --noEmit
  5. npm run build → dist/
  6. npx playwright install chromium
  7. Start backend in background; poll /api/health for 30s
  8. find tests/ -name "*.spec.ts" → npx playwright test (or warn)
  9. Kill backend
Rule of thumb: if npm run ci:local passes, your push will pass CI (modulo environment differences). It's the cheapest way to avoid "works on my machine, fails on CI" surprises.
